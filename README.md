# WebServer

Linux 下基于 C++14 实现的轻量级、高性能 Web 服务器，旨在学习和实践 Linux 网络编程、多线程并发模型以及底层系统调用。

---

## 功能特性

- 采用基于 **epoll** 模拟的 **Proactor** 并发模型，主线程负责全部的 I/O 读写操作，工作线程池专注处理业务逻辑，实现 I/O 与计算的解耦。
- 支持边缘触发 (ET) 和水平触发 (LT) 模式，通过设置非阻塞 socket 提高并发处理能力。
- 实现基于 **有限状态机 (FSM)** 的 HTTP 协议解析器，支持 HTTP/1.1 长连接 (Keep-Alive)，支持 GET / POST 请求解析。
- 采用 **mmap** 内存映射与 **writev** 聚集写技术，减少 CPU 拷贝和内存开销，实现高效文件传输。
- 构建基于 **最小堆 (Min-Heap)** 的高精度定时器，结合哈希映射实现 O(1) 复杂度的节点查找与续命，统一管理超时连接，剔除死连接释放系统资源。
- 基于生产者-消费者模型实现 **线程安全阻塞队列**，并以此搭建单例模式下的 **异步日志系统**，解决日志落盘阻塞工作线程的问题。
- 屏蔽 Nagle 算法 (TCP_NODELAY) 降低网络响应延迟，配置 SO_LINGER 实现连接的优雅关闭。

## 编译与运行

项目提供了快速构建脚本，或者可以直接使用 Makefile 进行编译。

### 方式一：使用脚本一键构建

```bash
chmod +x build.sh
./build.sh
```

### 方式二：手动编译

```bash
make
```

### 启动服务器

```bash
# 默认端口 9006
./server

# 携带自定义参数启动
# -p: 端口号  -t: 线程池线程数量  -c: 日志开关 (0为开启，1为关闭)
./server -p 8080 -t 8 -c 0
```

### 瓶颈剖析：为什么单机极限 QPS 停留在 1.3w 左右？

在压测过程中发现，当并发连接数达到 4000 甚至 10000 时，QPS 稳定在 1.3w 上下无法继续线性增长，且伴有极少量的 Timeout 错误（最大延迟触及 1.8s）。经过源码级追踪与资源监控分析，当前架构的性能瓶颈主要集中在以下几点：

#### 1. 主线程 I/O 负担过重 (模型局限性)

本项目采用的是模拟 Proactor 模型。在代码实现中，所有的网络读取 (recv) 和发送 (writev) 操作均由 main 线程（即 epoll_wait 所在的事件循环主线程）同步完成。工作线程池仅负责解析 HTTP 报文和准备发送指针。在万级并发下，单核主线程频繁陷入系统调用处理海量的 I/O 事件，导致 CPU 跑满。accept 新连接和后续读写操作发生排队，进而引发部分客户端 Timeout 超时。

**优化方向：** 向主从 Reactor 模型 (Multi-Reactors) 或真正的异步 Proactor 模型演进，将 read/write 等耗时的 I/O 系统调用也下放给工作线程或 Sub-Reactor 处理，彻底解放主线程。

#### 2. 多线程锁竞争开销

为了保证线程安全，线程池 (`std::mutex m_queuelocker`) 以及异步日志模块均使用了互斥锁。在 1.3w 的极端高并发请求下，多个工作线程频繁竞争同一个任务队列的全局锁，导致了不容忽视的线程上下文切换和锁等待开销。

**优化方向：** 引入无锁队列 (Lock-Free Queue)，或者采用每个 Reactor 绑定独立线程和独立任务队列的方式，消除任务分发时的全局锁竞争。

#### 3. 压测环境的干扰

压测客户端 wrk 和 WebServer 运行在同一台宿主机上。wrk 产生万级并发连接及高频发包本身就需要消耗大量的 CPU 和内核网络栈资源，这与 WebServer 产生了直接的资源抢占，导致测试出的 QPS 低于服务器在完全独立物理机上的真实极限网络吞吐性能。
